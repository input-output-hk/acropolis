# Implementation Plan: Basic P2P Peer Discovery for PNI

**Branch**: `780-Basic-P2P-Peer-Client` | **Date**: 2026-03-09 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/780-Basic-P2P-Peer-Client/spec.md`

## Summary

Add cold peer set management, peer-sharing-based peer discovery, and periodic churn to the PNI module. The node seeds its cold peer list from static config on startup, promotes cold peers to hot connections as needed, queries connected hot peers for new addresses via the peer-sharing mini-protocol (using a separate short-lived TCP connection), and periodically rotates hot peers to sample the broader network. All behaviour is internal to the PNI module — no new inter-module messages, no persistence layer, no pallas modifications.

## Technical Context

**Language/Version**: Rust 2024 Edition (workspace)
**Primary Dependencies**: pallas 0.34 (Bearer, Plexer, handshake miniprotocol, minicbor), tokio (async, time::interval, time::timeout), tracing (structured logging)
**Storage**: N/A — cold peer list is in-memory only (HashSet<String>)
**Testing**: cargo test (unit tests for PeerManager logic; integration tests for peer-sharing client against mock TCP server)
**Target Platform**: Linux server (same as existing PNI)
**Performance Goals**: Peer-sharing exchange completes within `peer_sharing_timeout_secs` (default 10s); churn/discovery ticks are background tasks that do not block block synchronisation
**Constraints**: No pallas modifications; peer-sharing requires separate TCP connection per exchange (pallas 0.34 limitation — channels must be subscribed before plexer spawn); handshake must negotiate V11+ for peer-sharing capability
**Scale/Scope**: Cold peer list capped at 4 × target_peer_count (default 60 entries); at most one peer-sharing exchange per hot peer per 5 minutes

## Phase 0: Research

**Status**: Complete. See [research.md](research.md).

Key decisions resolved (see research.md for full rationale):

| ID | Decision |
|---|---|
| D-001 | Peer-sharing requires V11+ handshake — use `v11_and_above(magic)` |
| D-002 | Separate short-lived TCP connection per exchange (pallas 0.34 constraint) |
| D-003 | Manual CBOR via `pallas_codec::minicbor` 0.25.1 — 3 message types |
| D-004 | Cold peers stored as `HashSet<String>` with `"host:port"` string keys |
| D-005 | Timer integration via `tokio::select!` over `events.recv()` + two `interval` futures |
| D-006 | Full peer selection algorithm: filter by cooldown → random pick → `record_query()` BEFORE async exchange |
| D-007 | V11 handshake failure on old peer → log debug, skip, try another peer |
| D-008 | No warm peer tier — all active connections are hot; warm tier deferred |
| D-009 | Failed cold peers → session blacklist; hot peers use existing 5s-backoff reconnect unchanged |
| D-010 | Random eviction at cold cap — replaces "suppress peer-sharing at cap" |
| D-011 | Address validation at trust boundary in peer_sharing.rs; self-address not filtered |
| D-012 | Peers added to `peers` map at spawn time (not connect) — `min_hot_peers` is implicit concurrency cap |
| D-013 | Inbound connections out of scope — client-only, no TCP listener |
| D-014 | No last-seen timestamps — random eviction + session blacklist provide adequate freshness |
| D-015 | Discovery runs on every tick (continuous) — not gated on count below target |
| D-016 | No peer freshness timestamps — three existing mechanisms sufficient for basic P2P |
| D-017 | No additional canonicalization — peer-sharing produces canonical `"ip:port"` by construction |
| D-018 | No peer scoring — cold peers have no data to score; hot peer instrumentation out of scope |
| D-019 | No global rate limit — 60s tick is structural ceiling; no inbound surface exists |
| D-020 | Partition handling covered by D-012 (storm bound) + D-009 (hot peer reconnect loop) |
| D-021 | Response truncated at `amount` entries during CBOR decode (FR-014) — prevents memory amplification |

## Phase 1: Design

### Component Design

#### peer_manager.rs (new)

```
pub struct PeerManager {
    cold_peers:       HashSet<String>,
    failed_peers:     HashSet<String>,      // session blacklist for failed cold promotions
    sharing_cooldown: HashMap<PeerId, Instant>,
    config:           PeerManagerConfig,
}

impl PeerManager {
    pub fn new(config: PeerManagerConfig) -> Self
    pub fn seed(&mut self, addresses: &[String], hot: &HashSet<String>)
    pub fn take_cold_peer(&mut self) -> Option<String>
    pub fn mark_failed(&mut self, address: String)   // move to session blacklist
    pub fn add_discovered(&mut self, addresses: Vec<String>, hot: &HashSet<String>)
    // Filters: already in cold, already hot, in failed_peers
    // If cold_count >= cap: randomly evicts one existing cold peer before inserting
    // Eviction applied per-insertion (not batch), ensuring cap invariant holds at all times
    pub fn needs_discovery(&self, hot_count: usize) -> bool
    // Returns true iff peer_sharing_enabled && hot_count > 0
    // NOT gated on cold_count — discovery runs continuously (D-015)
    pub fn can_query(&self, peer_id: PeerId) -> bool
    pub fn record_query(&mut self, peer_id: PeerId)
    pub fn should_churn(&self, hot_count: usize) -> bool
    pub fn cold_count(&self) -> usize
}
```

#### peer_sharing.rs (new)

```
/// Request up to `amount` peer addresses from a remote node.
/// Opens a dedicated V11+ handshake connection, performs the peer-sharing
/// exchange, and returns discovered "ip:port" strings.
/// Returns Ok(vec![]) if the peer does not support V11+.
pub async fn request_peers(
    address: &str,
    magic: u32,
    amount: u8,
    timeout: Duration,
) -> Result<Vec<String>, PeerSharingError>
```

#### network.rs changes

- `NetworkManager` gains `peer_manager: Option<PeerManager>` (None when disabled)
- `NetworkEvent` gains `PeersDiscovered { from_peer: PeerId, addresses: Vec<String> }`
- `run()` restructured to `tokio::select!`:
  ```rust
  loop {
      tokio::select! {
          Some(event) = self.events.recv() => self.on_network_event(event).await?,
          _ = churn_ticker.tick(), if self.peer_manager.is_some() => self.on_churn().await,
          _ = discovery_ticker.tick(), if self.peer_manager.is_some() => self.on_discovery().await,
      }
  }
  ```
- `on_churn()`: if hot_count > min_hot_peers, demote random peer, log, trigger promotion check
- `on_discovery()`: if needs_discovery, pick eligible hot peer, spawn peer-sharing task, send PeersDiscovered event on completion
- Promotion check: called after any disconnect or demotion

### Logging (FR-012)

Structured tracing fields, levels as specified:
```rust
// Promotions and demotions (info)
info!(address = %addr, hot = hot_count, cold = cold_count, "peer promoted cold→hot");
info!(address = %addr, hot = hot_count, cold = cold_count, "peer demoted hot→cold (churn)");

// Peer-sharing outcomes (info / warn)
info!(peer = %peer_id, discovered = addrs.len(), cold = cold_count, "peer-sharing complete");
warn!(peer = %peer_addr, error = %e, "peer-sharing exchange failed");
debug!(peer = %peer_addr, "peer-sharing skipped: V11 handshake not supported");

// Cold peer promotion failure (warn)
warn!(address = %addr, error = %e, "cold peer promotion failed; adding to session blacklist");

// Cold peer eviction at cap (info)
info!(evicted = %evicted_addr, cold = cold_count, "cold peer evicted (cap reached)");

// Peer counts (info — emitted after each churn/promotion/discovery event)
info!(hot = hot_count, cold = cold_count, "peer counts");
```
